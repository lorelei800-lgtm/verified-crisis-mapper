# Fusion Pipeline — Webhook-driven Multi-Source Crisis Data Ingestion

> Multi-hazard public-data ingestion layer for Verified Crisis Mapper.
> Citizen reports remain the heart of the system; this pipeline adds three
> public sources so the dashboard is useful in the 0–2 hour window after a
> disaster, before citizens can physically file reports.

## Sources (all multi-hazard, all open)

| Source | Endpoint | Hazards covered | Method |
|---|---|---|---|
| **GDACS Alerts** | `https://www.gdacs.org/xml/rss.xml` | Earthquake, tsunami, cyclone, flood, volcano, drought | RSS poll, 5min |
| **Copernicus EMS Rapid Mapping** | `https://emergency.copernicus.eu/mapping/list-of-activations-rapid.json` | All hazards (satellite extent maps) | REST poll, 10min |
| **ReliefWeb (UN OCHA)** | `https://api.reliefweb.int/v1/reports` | All hazards (situation reports) | REST poll, 15min |

GDACS = Global Disaster Alert and Coordination System (UN-affiliated).
Copernicus EMS = European Commission's emergency mapping service.
ReliefWeb = the OCHA-run humanitarian information clearinghouse.

## How it fits

```
[Source Layer]                    [Fusion Pipeline]                [CMS]              [Dashboard]
GDACS RSS         ─┐
Copernicus REST   ─┼──→ normalize → H3 cluster → trust-score-v2 ──→ verified-events ──→ React PWA
ReliefWeb REST    ─┤    (CrisisEvent)  (FusedEvent)                                       (existing
[Citizen PWA]     ─┘                                                                      Dashboard +
                                                                                          Source Filter
                                                                                          + Lineage Card)
```

Citizen reports continue to flow through `damage-reports` (unchanged).
Webhook sources land in a new `verified-events` CMS model. The Dashboard
queries both and shows them on one map with a source filter.

## Running locally

```bash
cd demo
# Live fetch + post to CMS (requires VITE_CMS_TOKEN):
npm run fusion

# Dry run — fetch + normalize + dedupe + score, but don't POST:
npm run fusion -- --dry-run

# Refresh the static JSON the Dashboard reads (no CMS token needed):
npm run fusion:static

# Use a single source (faster, useful for debugging):
npm run fusion -- --source=gdacs
```

## How the data reaches the Dashboard

The pipeline writes to two destinations, used as fallbacks:

1. **Static JSON** at `public/verified-events.json` — generated by
   `npm run fusion:static`, committed to the repo, refreshed by the
   cron in `.github/workflows/fusion-cron.yml`. The Dashboard reads
   this via `src/services/verifiedEvents.ts`. This path needs **no**
   CMS write token, so it works on GitHub Pages out of the box.
2. **Re:Earth CMS `verified-events` model** — populated by
   `npm run fusion` when `VITE_CMS_TOKEN` is set. Future React code
   can read from here for live updates without rebuilding the static
   JSON, and the audit trail is queryable through the standard CMS UI.

## Re:Earth CMS `verified-events` model

To enable the live CMS path, create one model named `verified-events` in
your Re:Earth CMS project with the fields below. `post-to-cms.mjs` writes
exactly these keys; missing keys are silently dropped by the CMS.

| Key                 | Type    | Notes                                                                  |
|---------------------|---------|------------------------------------------------------------------------|
| `event_id`          | Text    | stable id, e.g. `gdacs-EQ-1541036` (recommend: unique constraint)      |
| `source_type`       | Select  | one of: `citizen`, `gdacs`, `copernicus`, `reliefweb`                  |
| `hazard_type`       | Select  | `earthquake`, `tsunami`, `cyclone`, `flood`, `volcano`, `drought`, `wildfire`, `other` |
| `title`             | Text    | one-line summary                                                       |
| `description`       | Text    | optional longer paragraph                                              |
| `lat`               | Number  | event latitude (decimal degrees)                                       |
| `lng`               | Number  | event longitude (decimal degrees)                                      |
| `h3_cell`           | Text    | H3 resolution-8 cell index                                             |
| `occurred_at`       | Text    | ISO 8601 timestamp of the event itself                                 |
| `detected_at`       | Text    | ISO 8601 timestamp when the source published the record                |
| `severity`          | Select  | `red`, `orange`, `green`, or empty                                     |
| `source_url`        | Text    | canonical link back to the source                                      |
| `trust_total`       | Number  | 0–100, sum of the four sub-scores below                                |
| `trust_source`      | Number  | 0–40 (Source/Content Integrity)                                        |
| `trust_geo`         | Number  | 0–30 (Geospatial Consistency)                                          |
| `trust_cross`       | Number  | 0–20 (Cross-Source Validation)                                         |
| `trust_meta`        | Number  | 0–10 (Source Reliability × Metadata)                                   |
| `source_count`      | Number  | distinct sources that fused into this event (1 = single-source)        |
| `is_fused`          | Boolean | `true` when `source_count >= 2`                                         |
| `lineage_json`      | Text    | JSON-encoded `{ fusedFrom: [{ sourceType, eventId, url, detectedAt }] }` |
| `affected_area_json`| Text    | JSON-encoded GeoJSON Polygon / MultiPolygon, or empty                  |

To use a different model name set `VITE_CMS_VERIFIED_MODEL=your-model-name`
in `.env`.

## Scheduling

`.github/workflows/fusion-cron.yml` runs `npm run fusion:static` every hour
on the main branch and commits the regenerated JSON when it differs. All
APIs we hit are unauthenticated and rate-friendly at this frequency.

## Trust Score v2

Same 4-factor weighting as citizen reports (40 / 30 / 20 / 10); applied across
all sources:

| Factor | Citizen interpretation | Webhook interpretation |
|---|---|---|
| Source / Content Integrity (0–40) | C2PA + EXIF + AI fingerprint | HTTPS signature + known publisher prior |
| Geospatial Consistency (0–30) | GPS accuracy band + area containment | Polygon coverage / coordinate plausibility |
| Cross-Source Validation (0–20) | H3 neighbours agree | Same H3 cell ±30min from another source |
| Source Reliability × Metadata (0–10) | Channel prior + landmark completeness | GDACS=10, Copernicus=10, ReliefWeb=9, Citizen prior |

Routing thresholds (≥80 green, ≥50 amber, else red) are unchanged.
