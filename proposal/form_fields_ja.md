## Field 1 — 問題と機会

大規模災害発生後の72時間において、人道支援担当者は不完全・遅延・地理的に不正確なデータに基づいて資源配分の意思決定を迫られます。衛星画像はマクロレベルの分析を提供しますが、個々のインフラ被害の把握は困難です。現地の人的報告は不可欠ですが、このデータをリアルタイムで収集・検証・可視化するスケーラブルで低摩擦なシステムは現在存在しません。

その結果は予測可能です：支援は可視化された・アクセス容易なエリアに集中し、同様に被害を受けながらマッピングされていない地域は支援が届かないまま取り残されます。2010年ハイチ地震では、体系的な被害データの不在が数週間にわたる資源配分の誤りを招きました。2016年熊本地震では、非公式なメッセージグループが事実上の報告システムとなりましたが、それは機能的であるものの非構造的・非検証的・公式調整の外側に存在するものでした。

**新たな未対処の脅威：AI生成の偽情報**

既存の危機報告ツールが対処できていない重大な新興課題が、生成AI時代におけるクラウドソースデータの完全性です。広く利用可能なAI画像・動画合成ツールを使えば、被害画像をほぼゼロコストで精巧に捏造でき、自動システムと人間のレビュアー双方を欺くのに十分なリアリティを持ちます。こうした画像が報告プラットフォームに投稿されることで、支援を被災していないエリアに誤誘導したり、真に被災した地域への支援を阻害したりする可能性があります。

組織的な大量報告キャンペーンによって被害優先度ランキングが政治的な目的で操作されることもあります。GPS偽装が緊急資源を誤った場所に誘導します。過去の災害から流用された画像が活発な危機の最中にSNSで広く拡散します。KoboToolbox・OpenStreetMap Tasking Manager・Ushahidiを含むいかなる既存の危機報告ツールも、AI生成コンテンツに対処できる系統的な自動検証機能を提供していません。

**誠実なスコープ定義**

本システムの運用環境を、意図的な透明性をもって定義します：

- **自然災害（地震・洪水・暴風雨・山火事）：** 対象。政府インフラが維持；PWA事前インストールとSMS/ラジオによるURL配布の両方が有効。
- **政府がネットを遮断しない紛争：** 対象。部分的なインフラがPWAおよびWhatsAppベースの報告をサポート。
- **政府によるインターネット完全遮断：** 明示的にスコープ外。いかなるウェブベースツールも完全遮断では機能しない。Direct-to-Cell（D2C）衛星接続がPhase 4の未来経路として位置づけられている。

この誠実なスコープ定義は設計原則そのものです。限界を明確に定義したシステムは、過度な約束をするシステムよりも意思決定者に信頼されます。

## Field 2 — ソリューション概要

**Verified Crisis Mapper**は、Re:Earth（EukaryaのプロダクショングレードなオープンソースWebGIS）上に構築された三層構造のオープンソースプラットフォームです。危機に直面したコミュニティが位置情報付き被害報告を投稿し、信頼性が自動検証されてリアルタイムで可視化されます。

**Layer 1 — データ収集**

三つのアクセス経路で最大限のリーチを確保：

- **Route A（事前インストール済みPWA）：** 防災訓練中に住民がQRコードをスキャンしてProgressive Web Appをインストール。危機時にはホーム画面アイコンをタップ → GPS自動取得で写真撮影 → 3タップフォーム → 送信。オフラインファースト：投稿はService Worker経由でローカルにキューされ、Background Sync経由で接続回復時に自動送信。
- **Route B（初回ブラウザアクセス）：** 緊急対応担当者がSMS・ラジオ・避難所の掲示でURLを配布。住民がモバイルブラウザからフォームに直接アクセス。
- **Route C（WhatsApp）※Phase 2実装予定：** ユーザーが専用WhatsApp Business API番号に写真を送信。ボットが被害分類プロンプト（1：軽微 / 2：部分的破損 / 3：全壊）を返し、画像のGPSメタデータを取得してシステムに報告を取り込む予定——アプリインストール不要。

各投稿で収集するデータ：写真・被害分類（軽微 / 部分的破損 / 全壊）・インフラ種別（8区分：住宅・商業・政府・公益・交通通信・コミュニティ・公共スペース・その他）・危機の性質（自然災害 / 技術的事故 / 人為的）・がれき状況評価・GPS座標・テキストによるランドマーク説明（GPS代替）。

**Layer 2 — Trust Verification Engine**

各報告に信頼スコア（0〜100点）を自動付与：

| 要素 | 配点 |
|---|---|
| 画像整合性（全投稿にAI生成検出を適用；EXIF GPS/タイムスタンプ整合性；対応デバイスのみC2PA暗号化検証を高信頼ボーナスとして適用） | 40点 |
| 地理空間整合性（衛星被害確率との照合） | 30点 |
| 報告相互検証（H3空間クラスタリング、約105mセル；報告初期の疎なフェーズではニュートラルスコア） | 20点 |
| 投稿メタデータ（タイムスタンプ・GPS精度・チャネル重み付け） | 10点 |

ルーティング：≥80 → マップ表示（緑）\| 50〜79 → 要注意表示（黄）\| <50 → 人的審査キュー（赤）。エンジンは物理的・地理的整合性のみを評価——政治的言論やユーザー身元は対象外。スコアリングモデルはC2PA不要で機能するよう明示的に設計され、ローエンドデバイスとWhatsApp投稿（転送中にメタデータが除去される）を平等に扱います。

**Layer 3 — 可視化・意思決定支援**

Verified Crisis MapperダッシュボードはReact PWAとMapLibre GL JS衛星画像で構築され、信頼度別カラーコードの被害ピン（緑/黄/赤）・優先エリアランキング・報告ごとのTrust Scoreブレークダウンを表示します。現場報告者が使うPWAがオペレーターダッシュボードを兼ねており、追加インストール不要。データはGeoJSON・CSV・Shapefile・REST APIでエクスポート可能——HDX・OCHA IM Toolbox・KoboToolbox・WFPシステムと互換。

**ライブプロトタイプ：** https://lorelei800-lgtm.github.io/verified-crisis-mapper/demo/
*（デプロイメント：東京洪水対応、神田川流域 — 千代田区・神田エリア）*

本プロトタイプは完全動作状態です：写真・GPS・Trust Scoreフィードバック付きの被害報告投稿；カラーコードの信頼度ピンを持つMapLibre GL JS衛星ダッシュボード；PINロックアウトと3ボタンレビュー（承認 / ↩保留に戻す / 拒否）を持つAdmin Review Panel；ダッシュボードに埋め込まれたスタッフログインボタン；Re:Earth CMSへの書き戻しと30秒クロスデバイス同期。市民はPINゲートなしでダッシュボードに無料アクセス可能。

**ゲーミングなしのエンゲージメント**

参加インセンティブは非金銭的かつゲーミング防止設計：送信直後の即時マップ確認（「あなたの報告が地図に表示されました」）・報告者自身が確認できるTrust Scoreフィードバック・Phase 0のコミュニティ訓練での事前登録による事前の当事者意識醸成。高速連続投稿・AI生成画像・GPS偽装はTrust Scoreエンジンによってアルゴリズム的にペナルティが科せられます。

国連の6公用語をすべてサポート：アラビア語・中国語・英語・フランス語・ロシア語・スペイン語。

**建物フットプリント連携**

React PWAダッシュボードにはOpenStreetMapおよび利用可能な場合は政府の地籍データから取得した建物フットプリントオーバーレイを組み込んでいます。報告者がマップ上の建物アウトラインをタップして位置フィールドを自動入力できる——密集した都市環境でのGPSの曖昧さを解消し、インフラの誤特定報告を削減します。フットプリントデータが利用不可能な場合は、テキストによるランドマーク説明（通り名・近隣施設・距離基準）が明確な位置取得のためのフォールバックとなります。

**報告バージョニング**

同一建物に対する複数の被害報告はデータベースでバージョン管理されます。ダッシュボードは最新の評価（現在の状況）と完全な履歴記録を並べて表示——オペレーターが時系列での悪化を追跡し、複数報告者からの独立した裏付けを確認し、相矛盾する評価を特定して人的レビューの対象にすることができます。

**モジュラー付加質問**

コア被害分類に加えて、フォームは状況把握の深化のためのオプションのモジュラーフィールドをサポートします：

- **電力インフラ：** 報告地点またはその周辺で電力は利用可能ですか？
- **医療サービス：** 周辺エリアで医療施設は機能していますか？
- **コミュニティニーズ：** 現地で最も急迫している必要物（食料・水・医療・避難場所・生計・WASH・インフラ・保護・行政支援・その他）

これらのフィールドは迅速な災害後報告での摩擦を最小化するためデフォルトでオプションですが、特定のデプロイメントコンテキストやUNDPの運用要件に応じて必須フィールドとして設定可能です。

**ダッシュボード意思決定支援フロー**

Verified Crisis Mapper React PWAダッシュボードは、政府・UNDP担当者に以下を提供します：

1. **リアルタイムフィード（ライブ実装済み）：** MapLibre GL JS衛星マップ上に信頼度別カラーコードのピン（緑/黄/赤）で投稿をリアルタイム表示。デスクトップサイドバーとモバイルボトムシートによる信頼度/インフラ種別フィルタリング・最新順/Trust Scoreソート・報告ごとのTrust Scoreブレークダウン。
2. **Admin Review Panel（ライブ実装済み）：** プログレッシブロックアウト付きPIN認証のオペレータービュー。承認 / ↩保留に戻す / 拒否の三ボタンレビューワークフロー——全決定が30秒以内に全デバイスに反映。ダッシュボードからスタッフログインボタンで直接アクセス。
3. **優先エリア表示（Phase 2）：** 報告密度・信頼スコアの高いエリアを強調するH3ヘキサゴナルクラスターオーバーレイ——ショートリスト選出後の完全実装を予定。
4. **エクスポートキュー（Phase 2）：** GeoJSON・CSV・Shapefile・REST API形式での構造化エクスポート——OCHA・WFP・HDX・KoboToolboxワークフローと直接互換。

## Field 5 — リスク分析

**リスク1：AI生成の偽情報・SNS誤情報（主要リスク）**

生成AIにより、説得力のある被害画像がほぼゼロコストで作成可能です。組織的な偽報告の投稿で支援が誤誘導されたり、SNSで拡散するリサイクル・AI生成画像が真正な報告が処理される前に人的審査容量を圧迫したりする可能性があります。

*対策：* Trust Score EngineがPrimary Defenseです。C2PA（Content Credentials）暗号化メタデータ検証によりデバイスの真正な出所を欠く画像を検出します。AI生成フィンガープリントパターンマッチングにより合成コンテンツをフラグします。H3空間クラスタリングにより各報告を近隣の投稿と照合——地理的な裏付けのない捏造報告は自動的に評価が下げられます。地理空間整合性チェックが報告位置を衛星由来の被害確率マップと照合します。この多層的な自動検証により、人的審査が介入する前にアタックサーフェスを削減します。

**リスク2：インターネット接続障害**

部分的・断続的な接続は災害環境で常態です。完全なネット遮断は明示的にスコープ外です。

*対策：* PWA Service Workerが報告フォームをデバイス上にキャッシュします。Background Sync APIがローカルに投稿をキューし、接続が回復した際に自動送信——ユーザー操作不要。WhatsAppはアプリのオフライン状態でネイティブにキューします。

**リスク3：C2PA非対応デバイス**

ハードウェアレベルのC2PA署名はすべてのスマートフォンモデルで普及しているわけではありません。

*対策：* C2PAは100点中40点に寄与します。利用不可能な場合、残り60点（衛星・H3・メタデータ）が意味のある検証を提供します。信頼度の低下は黄/赤ルーティングによって透明に伝えられます——保守的な結果が正しい動作です。

**リスク4：報告者の安全とプライバシー**

紛争隣接の文脈では、報告者の身元や位置情報が機密となる場合があります。

*対策：* 報告者の連絡先情報は完全に任意。アカウント作成不要。GPSデータは保存時に暗号化。役割ベースアクセスにより報告者は自分の投稿確認のみ閲覧；集計データは認証済みオペレーターアカウントのみにアクセス可能。データ保持ポリシーはデプロイメント単位で設定可能。

**リスク5：GPSスプーフィングと位置詐称**

悪意ある行為者が資源を誤誘導するために座標を偽装する可能性があります。

*対策：* 地理空間整合性スコアリングが報告座標を衛星被害解析と照合します。H3外れ値検出が周辺クラスターパターンと矛盾する位置の報告をフラグします。

**リスク6：スケール時のインフラコスト**

*対策：* クラウド非依存のDockerデプロイにより特定ベンダーへの依存を回避します。$50,000賞金がPhase 1〜2の開発をカバーします。Phase 3+の運用コストはUNDPパートナーシップまたは政府ホスト型デプロイで対応します。
